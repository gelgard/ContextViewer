ARCHITECTURE RULES

1. Architecture-first workflow is mandatory.
2. Recovery files are part of the active source-of-truth layer.
3. AI task execution must remain structured and deterministic.
4. The architecture set includes machine-readable context projection files.
5. The command "обнови архитектурные файлы" must synchronize:
   - architecture files
   - plan files
   - recovery files
   - contextJSON/json_spec.md
   - a new populated contextJSON/json_<timestamp>.json
6. contextJSON/json_<timestamp>.json must contain aggregated and structured project context and must not be empty.
7. json_spec.md must be overwritten whenever JSON schema or generation logic changes.
8. The latest timestamped JSON snapshot is the authoritative runtime source for the external visual application.
9. The visual application must use the latest JSON for structure, statuses, roadmap, graph, and summaries.
10. Markdown files may be opened by path for raw content display only and must not be required to compute project state.
11. Context JSON must be sufficient for an external visualization application to render:
   - architecture tree
   - file purposes and update rules
   - architecture dependency graph
   - plan status
   - AI task status
   - roadmap
   - executive summary
   - customer status summary
   - last iteration summary
12. After a task is completed and verified, the assistant must provide commit text for that task.
13. Test instructions must be concise, informative, and written as explicit step-by-step actions.
14. Test instructions must avoid general phrases and visual actions and must explicitly list what the user should send back for validation.
15. Every test step must specify the exact execution method, including command, SQL, input, or callable entry point; abstract verbs without execution details are invalid.
16. On each Stage transition, the assistant must explicitly announce the new Stage before issuing the next AI task.
17. Stage transition requires git flow: merge current branch into `development`, then create `feature/stage<stageNum>`.
18. Stage transition instructions must include exact reproducible git commands.
19. Architecture sync workflow is archive-first:
   - first notify that sync is needed
   - then receive current project archive
   - then perform full sync
   - only after that execute "обнови архитектурные файлы"
20. Architecture sync is incomplete without regenerated contextJSON outputs.
21. Command "дай следующую AI task" is invalid unless `ai_tasks/NNN_*.md` is physically created before response.
22. AI task response must include explicit created-file marker: `AI Task file created: /ai_tasks/NNN_*.md`.
23. Pre-response self-check is mandatory: file existence, numbering continuity, stage/substage sync, executable tests.
24. If AI task file is missing, assistant must stop normal output and emit only: `BLOCKED: AI task file missing, creating it now.` then create the file.
25. After user submits test results, assistant must auto-generate "List of changed files" from `git status --short` and validate file scope against current AI task.
26. Changed-files validation uses task commit boundary assumption: expected changes belong to current task; unrelated changes must be flagged explicitly.
27. Changed-files validation must exclude transient runtime artifacts (for example `.tmp_pg_*/pg_stat_tmp/*`).
28. For command "дай следующую AI task", the response format is a hard gate and must contain, in strict order:
   - `AI Task file created: /ai_tasks/NNN_*.md`
   - `Cursor prompt (EN)`
   - `Manual Test (exact commands)`
   - `What to send back for validation`
29. If block `Manual Test (exact commands)` is missing, assistant must emit only:
   - `BLOCKED: response format violation, regenerating with full test section.`
30. On any next-task response-format violation, assistant must immediately regenerate a fully compliant response before any further output.
