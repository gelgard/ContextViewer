ARCHITECTURE RULES

1. Architecture-first workflow is mandatory.
2. Recovery files are part of the active source-of-truth layer.
3. AI task execution must remain structured and deterministic.
4. The architecture set includes machine-readable context projection files.
5. The command "обнови архитектурные файлы" must synchronize:
   - architecture files
   - plan files
   - recovery files
   - contextJSON/json_spec.md
   - a new populated contextJSON/json_<timestamp>.json
6. contextJSON/json_<timestamp>.json must contain aggregated and structured project context and must not be empty.
7. json_spec.md must be overwritten whenever JSON schema or generation logic changes.
8. The latest timestamped JSON snapshot is the authoritative runtime source for the external visual application only.
9. The visual application must use the latest JSON for structure, statuses, roadmap, graph, and summaries.
10. Markdown files may be opened by path for raw content display only and must not be required to compute project state.
10a. `contextJSON/json_<timestamp>.json` must never define project architecture, planning policy, testing policy, execution policy, or development methodology.
10b. Validation JSON artifacts are a separate class of machine-readable outputs and must not be conflated with `contextJSON` exports.
10c. Higher-level validation may consume validation artifacts as execution evidence, but may not treat `contextJSON` as architecture authority.
11. Context JSON must be sufficient for an external visualization application to render:
   - architecture tree
   - file purposes and update rules
   - architecture dependency graph
   - plan status
   - AI task status
   - roadmap
   - executive summary
   - customer status summary
   - last iteration summary
12. After a task is completed and verified, the assistant must provide commit text for that task.
13. Test instructions must be concise, informative, and written as explicit step-by-step actions.
14. Test instructions must avoid general phrases and visual actions and must explicitly list what the user should send back for validation.
15. Every test step must specify the exact execution method, including command, SQL, input, or callable entry point; abstract verbs without execution details are invalid.
15a. Hard anti-duplication rule: the local agent layer owns architecture planning, architecture synchronization, validation logic, changed-file review, next-step planning, and user-facing orchestration.
15b. Cursor is used only for code-writing, implementation edits, development execution, and code-adjacent documentation changes required by the current AI task.
15c. The `Cursor prompt (EN)` block must not duplicate local-agent responsibilities or repeat content already expressed in `Manual Test (exact commands)` or `What to send back for validation`.
16. If a task affects UI, frontend, HTML preview, browser output, or any visual product surface, the test section must also include a dedicated visual manual-test subsection with explicit viewing steps and exact visual evidence required for validation.
16a. For each UI task, Layer 4 visual validation is mandatory and non-substitutable:
   - Codex-safe HTML marker checks are required;
   - user-side manual visual validation via explicit step-by-step scenario is required.
16b. Playwright may be used as optional diagnostic tooling for UI discrepancies, but is not part of the default closure path.
16c. Fast smoke execution policy is mandatory by default:
   - first run the highest-level stage gate script;
   - run underlying child smoke scripts only for failure localization, diagnostics, or explicit user request;
   - avoid duplicate re-runs of unchanged heavy smoke scripts in one validation cycle.
16c1. Validation profile lock is mandatory:
   - `fast` mode is the required default for routine acceptance validation;
   - `full` mode is allowed only when:
     - explicitly requested by the user, or
     - `fast` mode failed and deeper diagnostics are required;
   - proactive `full` runs as default behavior are prohibited.
16c2. Acceptance gating mode policy is mandatory:
   - `fast` mode is the authoritative acceptance gate for UI/stage orchestration validation.
   - `full` mode is diagnostic-only and non-blocking for closure when:
     - `fast` gate already passed in the same code/validation cycle, and
     - no new code changes were introduced after the passing `fast` evidence.
   - `full` diagnostic failures must be surfaced explicitly and must not silently override a valid `fast` acceptance result.
16d. Anti-hang validation policy is mandatory:
   - do not run heavy UI smoke scripts concurrently on the same local port;
   - if concurrent execution is unavoidable, each process must use a distinct port;
   - heavy UI validations are sequential by default unless explicitly split for diagnostics.
16d1. Interrupted-run cleanup is mandatory:
   - before each heavy validation cycle, verify and terminate stale validation/server processes from prior aborted runs;
   - after user interruption, no new heavy run may start until cleanup + process-state verification is complete.
16e. Timeout/diagnostic policy for heavy validations:
   - long-running smoke/gate commands must be executed with bounded timeout guidance;
   - on environments without `timeout`, use a python timeout wrapper;
  - timeout events must be treated as diagnostic failures and followed by targeted reruns.
16e1. Step-level watchdog diagnostics are mandatory:
   - each heavy child step in orchestration scripts must include bounded execution.
   - on timeout, output must include explicit `timeout_step` classification (for example: `delivery`, `handoff`, `readiness`, `secondary_gate`).
   - timeout diagnostics must include blocker classification (`benchmark_leg_timeout`, `env_network`, `port_process_hygiene`, `contract_logic`).
17. If a task generates prompts for an external Figma/design workflow, the test section must specify the exact prompt blocks and the exact returned design artifacts required for validation.
18. If a task validates returned Figma/design results, the test section must specify the exact visual artifacts to return and the exact structural/visual checks that must pass.
19. The validated Stage 8 preview / handoff state is a preserved checkpoint and must not be replaced by the Figma design branch.
20. The Stage 8 Figma design branch refines implementation and architecture decisions without replacing recovery/planning authority or the original implementation path.
21. Prompt generation, design validation, Figma import, and post-Figma implementation refinement must execute through numbered AI tasks only.
22. After an approved Figma artifact is imported into the workspace, architecture files, plans, recovery files, and the external viewer export must be synchronized before further implementation.
23. On each Stage transition, the assistant must explicitly announce the new Stage before issuing the next AI task.
24. Stage transition requires git flow: merge current branch into `main`, then create `feature/stage<stageNum>`.
25. Stage transition instructions must include exact reproducible git commands.
26. Architecture sync workflow is workspace-first:
   - first use the current project workspace as the sync source
   - then perform full sync directly in workspace
   - if workspace access is unavailable, request current project archive and run archive-based fallback
   - then execute "обнови архитектурные файлы"
27. Architecture sync is incomplete without regenerated contextJSON outputs.
28. Command "дай следующую AI task" is invalid unless `ai_tasks/NNN_*.md` is physically created before response.
29. AI task response must include explicit created-file marker: `AI Task file created: /ai_tasks/NNN_*.md`.
30. Pre-response self-check is mandatory: file existence, numbering continuity, stage/substage sync, executable tests.
30a. Every new AI task must include a mandatory manager-facing non-technical step summary:
   - explain what will be done in the current step;
   - explain it from the user point of view, not from the programmer point of view;
   - explain why the step matters for the user or decision-maker;
   - avoid specialist/programming terminology;
   - keep it concise.
31. If AI task file is missing, assistant must stop normal output and emit only: `BLOCKED: AI task file missing, creating it now.` then create the file.
32. After user submits test results, assistant must auto-generate "List of changed files" from `git status --short` and validate file scope against current AI task.
33. Changed-files validation uses task commit boundary assumption: expected changes belong to current task; unrelated changes must be flagged explicitly.
33a. Commit-boundary separation is mandatory:
   - functional task files, architecture-sync files, and external viewer export files must be treated as distinct change classes.
   - `contextJSON/*` belongs to the external viewer export class and is not automatically part of a functional task commit.
   - if one execution cycle modifies both functional task files and architecture/export files, assistant must propose split commit boundaries before closure.
34. Changed-files validation must exclude transient runtime artifacts (for example `.tmp_pg_*/pg_stat_tmp/*`).
35. For command "дай следующую AI task", the response format is a hard gate and must contain, in strict order:
   - `AI Task file created: /ai_tasks/NNN_*.md`
   - `Manager Summary (non-technical)`
   - `Cursor prompt (EN)`
   - `Manual Test (exact commands)`
   - `What to send back for validation`
35a. Block `Manager Summary (non-technical)` is mandatory:
   - it exists for a manager or stakeholder without programming experience;
   - it must state what will be improved in this step and why that matters for the user;
   - it must use plain language and avoid technical jargon.
36. Block `Cursor prompt (EN)` must contain the prompt inside exactly one fenced code block so the UI exposes a `Copy` option.
36a. `Cursor prompt (EN)` must contain implementation instructions only and must not include:
   - `Manual Test`
   - `What to send back for validation`
   - architecture update steps
   - next-step planning
   - changed-files validation instructions
   - local-agent validation/orchestration instructions
37. If block `Manual Test (exact commands)` is missing, assistant must emit only:
   - `BLOCKED: response format violation, regenerating with full test section.`
38. On any next-task response-format violation, assistant must immediately regenerate a fully compliant response before any further output.
39. Every AI task must be explicitly mapped to product-goal Requirement IDs defined in `docs/plans/product_goal_traceability_matrix.md`.
40. If an AI task has no Requirement ID mapping, it is out of scope and must be blocked until mapping is added.
41. Task acceptance must include evidence that mapped Requirement IDs were actually validated by executable tests.
42. Context Restore Policy is mandatory and must be enforced before task execution.
43. Source priority for restore is strict:
   - project_recovery/*
   - AGENTS.md
   - docs/architecture/*
   - docs/plans/*
   - ai_tasks/*
   - contextJSON/json_<latest>.json
   - code/* (validation only)
44. Fast restore is required before EVERY new AI task.
45. Fast restore output must be a one-line summary only:
   - `FAST RESTORE: stage=<...> | task=<...> | next=<...> | gate=<...> | readiness=<ready|blocked>`
46. Full restore is mandatory after:
   - `обнови архитектурные файлы`
   - merge/stage transition
   - suspected desync
   - long pause
   - explicit command `обнови полный контекст`
   - Figma artifact import or design-sync update
47. Command mapping:
   - `обнови контекст` => Fast restore (default)
   - `обнови полный контекст` => Full restore (forced)
48. Long pause is defined as:
   - inactivity >= 4 hours OR
   - new calendar day since last restore OR
   - context handoff between agents/users
49. Blocked policy response format:
   - `BLOCKED: Context restore policy violation.`
   - `REQUIRED FIX: Run <Fast|Full> restore and resync required files.`
50. During validation, the assistant must always run `git status --short` itself and use it to generate changed-files scope review, even if the user did not provide that command output.
51. Every assistant response must start with a project/session marker line:
   - `ContextViewer-1 | <stage-or-topic>`
   - marker is mandatory to prevent cross-project chat confusion.
52. Validation pyramid is a permanent architecture rule:
   - Layer 1 = Codex offline unit checks only
   - Layer 2 = Codex offline contract/fixture checks only
   - Layer 3 = user or CI live integration checks
   - Layer 4 = user visual confirmation for HTML-preview tasks
53. Codex acceptance is limited to Layer 1 + Layer 2 evidence and must not depend on DB, HTTP server startup, curl, or live network access.
54. Layer 3 is mandatory for live closure evidence but is delegated to user/CI and must be routed through explicit manual-test commands with exported timeout profile.
55. Layer 4 is mandatory for every UI/HTML-preview task and must be satisfied through HTML marker checks plus user visual evidence; Playwright may supplement but does not replace user confirmation.
56. Child-script deduplication is a hard architecture invariant:
   - once a child verifier has produced JSON in a validation cycle, higher-level gates must consume that JSON instead of re-executing the child verifier
   - duplicate child invocation in the same unchanged validation cycle is a defect
57. Anti-loop execution policy is part of architecture, not only testing guidance:
   - identical failure twice with no code change => stop and classify
   - timeout beyond the declared layer ceiling => stop and classify
   - no blind retries against unchanged output
58. Lightweight validation operating model is mandatory:
   - one AI task must have exactly one primary acceptance gate
   - diagnostics are separate, explicitly labelled, and non-blocking by default
   - benchmark paths are diagnostic only unless explicitly declared as the single acceptance gate
59. Artifact-first validation is a permanent architecture rule:
   - heavy validation outputs must be treated as reusable JSON artifacts
   - higher-level gates must consume prior artifacts instead of recomputing heavy readiness/delivery/completion paths
   - artifact reuse is preferred over orchestration layering
60. Recursive orchestration is an architecture defect:
   - orchestration scripts must not invoke other orchestration scripts by default
   - a new wrapper over an existing wrapper is prohibited unless the lower layer is collapsed or replaced
61. Validation budgeting is an architecture constraint:
   - Layer 1 + Layer 2 target <= 30s, hard ceiling <= 60s total
   - primary Layer 3 integration target <= 60s, hard ceiling <= 120s
   - full task closure target <= 10 minutes including manual visual review
   - tasks that cannot fit this budget must be split or re-architected before execution
62. Timeout defaults are architecture-controlled:
   - 30s = DB-only / very small live checks
   - 60s = default single integration acceptance gate
   - 120s = benchmark / dual-leg diagnostics only
   - 420s = CI-only exceptional diagnostics and must never be the ordinary local default
63. Full restore must reconstruct the validation operating model, not just task/stage state:
   - report current validation architecture status
   - report migration status
   - report heavy legacy validation warnings
   - report next-step constraints under the lightweight model
64. Legacy heavy validators are classified as diagnostic-only legacy paths until replaced by artifact-first acceptance flows.
65. No future functional AI task may deepen the validation stack:
   - if a task would introduce another orchestration layer, it must first create or reuse an artifact-first acceptance path.
66. JSON-type separation is a hard architecture rule:
   - `contextJSON/*` = external viewer export only
   - validation artifacts = acceptance/diagnostic execution evidence only
   - neither JSON class may override `project_recovery/*`, `AGENTS.md`, or planning authority.
