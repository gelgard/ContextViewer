RESPONSE FORMAT RULES

- First line of every assistant response must include a project/session marker:
  - format: `ContextViewer-1 | <stage-or-topic>`
  - mandatory for all responses to prevent confusion across multiple active project chats
- If project context changes, assistant must update the marker in the very next response
- Always follow AI Task structure
- Always include terminal testing instructions when terminal testing is involved
- Always include:
  - changed files
  - full file content
  - test output
- Always include commit step after successful testing
- Always provide commit text after a task is accepted as completed
- Tests must be concise, informative, and describe each required step explicitly
- Test instructions must avoid general phrases and visual actions and must state exactly what to send back for validation
- Every test step must include the exact execution method; abstract verbs without executable instructions are invalid
- Hard anti-duplication rule: the local agent response owns planning, architecture, validation, changed-files review, and next-step orchestration
- Cursor is used only for code/development execution; the `Cursor prompt (EN)` block must contain implementation instructions only
- The `Cursor prompt (EN)` block must not repeat:
  - `Manual Test (exact commands)`
  - `What to send back for validation`
  - architecture synchronization instructions
  - changed-files validation instructions
  - next-step planning or local validation responsibilities
- If a task affects UI, frontend, HTML preview, browser output, or any visual product surface, tests must also include a dedicated visual manual-test section with explicit viewing steps and exact visual evidence to send back for validation
- For each UI task, both validation contours are mandatory:
  - Assistant must run Playwright and report:
    - steps executed
    - result per step (pass/fail)
    - visual mismatch list (if any)
  - User must also execute manual visual validation by explicit step-by-step scenario
  - Manual visual validation is mandatory and is not replaced by Playwright
- Fast smoke execution policy (mandatory by default):
  - run one top-level orchestration gate first (stage readiness/completion gate)
  - run underlying child smoke scripts only when:
    - top-level gate failed
    - failure localization is needed
    - user explicitly requests full decomposition
  - do not repeat expensive identical smoke runs in the same validation cycle without code changes
- Validation profile lock (mandatory):
  - use `fast` mode for routine acceptance checks by default
  - use `full` mode only if:
    - user explicitly requested `full`, or
    - `fast` failed and deeper diagnostics are required
  - do not launch `full` preemptively as a default validation route
  - acceptance policy:
    - `fast` is the authoritative closure gate
    - `full` is diagnostic/non-blocking when `fast` already passed in the same unchanged validation cycle
    - report `full` failures explicitly as diagnostics (do not silently override `fast` acceptance)
- Anti-hang test-instruction policy (mandatory):
  - instructions must avoid parallel heavy UI smoke runs on the same port
  - if parallel runs are required, each command set must use a unique port
  - heavy validation commands should be sequential by default
  - for potentially long commands, include bounded-time execution guidance and fallback diagnostics
  - if a command appears stalled, instructions must include safe stop + diagnostic rerun steps
  - before any new heavy run, assistant must verify and clean stale processes from interrupted prior runs
  - after user interruption, assistant must report process-state check result before restarting heavy validation
  - timeout diagnostics must include explicit `timeout_step` classification and blocker class
- If a task generates prompts for an external Figma/design workflow, tests must include the exact prompt blocks to use and the exact returned artifacts required for validation
- If a task validates returned Figma/design results, tests must include the exact artifacts to return and the exact structural and visual checks that must pass
- On Stage transition, always announce the new Stage and provide exact git commands to merge into `development` and create `feature/stage<stageNum>`
- For command "дай следующую AI task", the response is valid only if `ai_tasks/NNN_*.md` was created before sending
- AI task response must include: `AI Task file created: /ai_tasks/NNN_*.md`
- For command "дай следующую AI task", response must use strict block order:
  1) `AI Task file created: /ai_tasks/NNN_*.md`
  2) `Cursor prompt (EN)`
  3) `Manual Test (exact commands)`
  4) `What to send back for validation`
- Block `Cursor prompt (EN)` must contain the prompt inside exactly one fenced code block so the UI exposes a `Copy` option
- Missing block `Manual Test (exact commands)` is a hard failure; output only:
  - `BLOCKED: response format violation, regenerating with full test section.`
- On next-task format violation, assistant must immediately regenerate full compliant response before any other output
- After user sends test results, assistant must generate `List of changed files` automatically from `git status --short`
- Assistant must run `git status --short` itself during validation even if the user did not include it
- Assistant must validate that changed files belong to current AI task scope and explicitly flag unrelated changes
- Changed-files auto-check must ignore transient runtime artifacts (for example `.tmp_pg_*/pg_stat_tmp/*`)
- Before sending AI task response, run self-check for:
  - file existence in `ai_tasks/`
  - numbering continuity
  - stage/substage sync
  - executable test steps with exact commands
  - Requirement ID mapping to `docs/plans/product_goal_traceability_matrix.md`
- If self-check fails because file is missing, output only:
  - `BLOCKED: AI task file missing, creating it now.`
- If self-check fails because Requirement ID mapping is missing, output only:
  - `BLOCKED: AI task has no product-goal mapping, add Requirement IDs first.`
- Context restore gate:
  - before every new AI task, execute Fast restore
  - if Full restore trigger occurred, Fast restore alone is invalid
  - Fast restore response must be one-line only:
    `FAST RESTORE: stage=<...> | task=<...> | next=<...> | gate=<...> | readiness=<ready|blocked>`
- Command mapping for restore:
  - `обнови контекст` => Fast restore (default)
  - `обнови полный контекст` => Full restore (forced)
- Full restore triggers:
  - after `обнови архитектурные файлы`
  - after merge/stage transition
  - when desync is suspected
  - after long pause
  - after Figma artifact import or design-sync update
- Long pause rule:
  - inactivity >= 4 hours OR
  - new calendar day since last restore OR
  - context handoff between agents/users
- On Context Restore Policy violation, output only:
  - `BLOCKED: Context restore policy violation.`
  - `REQUIRED FIX: Run <Fast|Full> restore and resync required files.`
- Do not skip numbering
- No large unstructured responses

Architecture sync rule:
When the user asks "обнови архитектурные файлы":
1. use current project workspace as the primary synchronization source
2. perform full sync directly in workspace
3. only if workspace is unavailable, request archive and perform archive-based fallback sync
4. then execute architecture synchronization
5. include:
   - contextJSON/json_spec.md
   - a new populated contextJSON/json_<timestamp>.json

The generated context JSON must contain aggregated and structured information, not an empty schema stub.

The newest timestamped JSON must be treated as the authoritative runtime source for the visual application.
Markdown files are referenced by path for raw content display only.
Visual Validation Rule (Layer 4, mandatory for every UI task):
- every AI task that creates or modifies HTML preview content must include a dedicated Visual Validation block in Manual Test
- Visual Validation block must contain two sub-steps:

  Sub-step A — HTML structure check (Codex-safe, < 5 seconds, no browser):
    grep -q 'render_profile="<expected>"' <output_dir>/index.html
    grep -q 'data-section="<section>"' <output_dir>/index.html  [repeat for each expected section]

  Sub-step B — Manual browser open (user, instant):
    open <output_dir>/index.html
    OR: python3 -m http.server <port> --directory <output_dir> & ; open http://localhost:<port>

- Visual evidence to send back: screenshot OR confirmed checklist of visible sections
- Visual validation is a separate evidence block; it does not replace Layer 3 integration evidence
- Playwright is optional for Layer 4; HTML grep + browser open is sufficient and always preferred
- Visual Validation block must appear as the last sub-section of Manual Test, clearly labelled "### Visual (Layer 4)"

Codex Acceptance Gate block (mandatory in every AI task response):
- every AI task response must include an explicit "Codex Acceptance Gate" block
- this block contains only Layer 1 + Layer 2 commands
- each command must include an expected exit code or expected output
- the block must be labelled "### Codex Acceptance Gate (offline, Layer 1+2)"
- commands in this block must not require live DB, HTTP server, or network
- maximum 7 commands per gate block; each must complete in < 10 seconds
